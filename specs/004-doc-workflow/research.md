## Issue Analysis and Resolution Decisions

### Issue 1: Case-Sensitivity Mismatch

**Problem**: The file on disk is `docs/architecture.md` (lowercase), but the workflow and templates reference `docs/ARCHITECTURE.md` (uppercase). On case-insensitive macOS this works, but on case-sensitive Linux CI (Ubuntu runners), these are two different files. The agent could create `docs/ARCHITECTURE.md` as a new file, leaving the stale `docs/architecture.md` in place.

**Current state**:
- On disk: `docs/architecture.md` (15,566 bytes, last modified Mar 2)
- In workflow scope step: `DOCS_TO_UPDATE="$DOCS_TO_UPDATE docs/ARCHITECTURE.md docs/API.md"`
- In template `07-conditional/architecture-api.md`: `#### ARCHITECTURE.md`

**Decision**: **Rename the file to uppercase** (`docs/ARCHITECTURE.md`) to match the workflow references. Rationale:
1. All other doc files use uppercase: `API.md`, `AUTOMATION.md`, `CONTRIBUTING.md`, `TESTING.md`
2. Uppercase is the project convention for documentation files
3. Renaming one file is simpler than updating all workflow/template references to lowercase
4. `git mv docs/architecture.md docs/ARCHITECTURE.md` handles this cleanly

**Implementation**:
```bash
git mv docs/architecture.md docs/ARCHITECTURE.md
```

### Issue 2: Three Expected Doc Files Never Exist

**Problem**: The workflow scope references `docs/ROUTING.md`, `docs/PROVIDERS.md`, and `docs/CONFIGURATION.md`, but these files have never been created. When the agent runs, it may attempt to `read` them (getting errors) or skip them.

**Current state**:
- `docs/ROUTING.md` — does not exist
- `docs/PROVIDERS.md` — does not exist
- `docs/CONFIGURATION.md` — does not exist

**Decision**: **Create minimal seed files** for each. The agent will then update them rather than create them from scratch. Seed files contain:
1. A title heading
2. A brief description
3. A "Generated by Alexi" footer
4. A comment instructing the agent to replace the content

This reduces the agent's work (update vs. create) and ensures `read` calls don't fail.

**Implementation**: Create three files in `docs/`:
```markdown
# Routing System

> This document is auto-generated by the documentation workflow. Content will be populated on the next PR that modifies routing-related code.

<!-- TODO: Agent should replace this content with actual routing documentation -->
```

### Issue 3: Three Unmanaged Doc Files in `docs/`

**Problem**: `docs/permission-system.md`, `docs/migration-guide-permission-prompts.md`, and `docs/permission-prompt-quick-ref.md` exist in `docs/` but are not in the workflow's scope. The current `git add docs/` stages everything in the directory, including these files.

**Decision**: **Replace `git add docs/` with explicit file staging**. Only stage the files listed in `$DOCS_TO_UPDATE`. This prevents the workflow from accidentally modifying or committing unmanaged files.

**Implementation**:
```bash
# Instead of: git add docs/ CHANGELOG.md || true
# Use:
for f in $DOCS_TO_UPDATE; do
  [ -f "$f" ] && git add "$f"
done
```

### Issue 6: Dual System Prompt (Inline + agenticChat Assembly)

**Problem**: The workflow provides an inline `--system` prompt AND `agenticChat()` builds its own assembled system prompt from agent prompts in `src/agent/prompts/`. This creates two system prompts that may conflict.

**Analysis**: Looking at `agenticChat.ts`, the `--system` flag content is prepended to the assembled system prompt. So both are used, but the inline one takes precedence for instructions that appear first.

**Decision**: **Simplify the inline system prompt to role-only**. Remove all detailed instructions from the inline `system-prompt.txt` (Mermaid requirements, file path instructions, documentation style) — these belong in the message prompt (assembled from templates). The inline system prompt should only set the agent's role:

```
You are a technical documentation writer for the Alexi project. Read source code files using the provided tools before writing documentation. Write to the exact file paths specified in the Documentation Scope section.
```

### Issue 7: Diff Preview Truncated to 100 Lines

**Problem**: `git diff ... | head -100` in the "Generate code diff summary" step truncates large diffs. The agent sees only a fraction of the actual changes.

**Decision**: **Remove the `head -100` truncation**. Instead:
1. Include the full `git diff --stat` (summary lines, not full patch) — this is already compact
2. For the detailed diff, include up to 500 lines. If the diff exceeds 500 lines, include the stat summary and add a note: "Full diff exceeds 500 lines. Use `read` and `grep` tools to examine source files directly."
3. More importantly: **update the prompt instructions** to tell the agent to always read source files using tools, not rely on the embedded diff.

**Implementation**: Replace `head -100` with `head -500` and add an overflow note. Update `04-diff-header.md` to include the instruction: "Note: This is a preview. Use the `read` and `grep` tools to examine actual source files for complete context."

### Issue 8: Overly Aggressive `.md` File Filtering

**Problem**: `grep -v '.*\.md$'` filters ALL markdown files from the changed-files list. This hides changes to:
- `AGENTS.md` — project coding guidelines
- `.github/templates/*.md` — prompt templates
- `README.md` — may contain meaningful changes
- `.specify/**/*.md` — speckit artifacts

**Decision**: **Filter only `docs/**/*.md` and `CHANGELOG.md`** from the changed files list. All other `.md` files should be treated as meaningful changes.

**Implementation**:
```bash
# Instead of: grep -v -E '.*\.md$'
# Use:
grep -v -E '^(docs/|CHANGELOG\.md$)' || echo ""
```

This keeps `AGENTS.md`, `.github/templates/*.md`, `README.md`, etc. in the changed files list.

### Issue 9: `--max-iterations 30` May Be Insufficient

**Problem**: For full regeneration with 8+ doc files, the agent needs to: (1) read each existing doc, (2) read relevant source files, (3) write/edit each doc. That's potentially 24+ tool calls minimum, plus exploratory `glob`/`grep` calls.

**Analysis**: With 30 iterations and the agent making 1-3 tool calls per iteration, the budget is approximately 30-90 tool calls. This should be sufficient for:
- 8 doc files × 2 calls each (read + write) = 16 calls
- ~10 exploratory reads (source files, glob, grep)
- Overhead = ~4 calls
- Total ≈ 30 calls, well within budget

**Decision**: **Keep `--max-iterations 30` for now**. The run that succeeded (ID `23112176253`) completed 41 tool calls within the iteration budget. If this becomes a problem, increase to 50. Add a warning in the PR comment if the agent hits the iteration limit.

### Issue 10: No Validation of Generated Docs

**Problem**: No markdown lint, Mermaid syntax check, or link validation after generation.

**Decision**: **Add lightweight validation as a separate workflow step** (post-generation, pre-commit). Validation failures produce warnings in the PR comment but do NOT block the commit. Tools:
1. `markdownlint-cli2` — already available via npx, no install needed
2. Mermaid syntax: use a regex check for unclosed code fences (lightweight; full Mermaid validation requires a headless browser, which is too heavy)
3. Link validation: skip for MVP (links are mostly internal and hard to validate in CI)

**Implementation**: Add a "Validate generated docs" step after the bot step and before the commit step. Run `npx markdownlint-cli2 docs/*.md` with `continue-on-error: true`. Capture output and include in PR comment.

### Issue 11: Commit Message Collision Risk

**Problem**: The workflow greps for the exact string `"docs: auto-generate documentation [skip ci]"` to find the last documentation commit. If a human uses this exact commit message, the workflow could incorrectly identify it as a bot commit.

**Decision**: **Use a more specific commit message with a machine identifier**. Change to:
```
docs: auto-generate documentation [skip ci] [alexi-bot]
```

And grep for `[alexi-bot]` tag instead of the full message. This is unlikely to collide with human commits.

## Additional Research: Workflow Step Ordering Bug

**Problem discovered during debugging**: The `force_flag` step runs AFTER `check_needed`, but `check_needed` references `steps.force_flag.outputs.force` which hasn't been set yet. GitHub Actions evaluates the expression as empty string (falsy), so force regeneration never works.

**Decision**: **Move `force_flag` step BEFORE `check_needed`**. This was already fixed in the `003-tui-polish` commit `3fe8861`, but needs to be verified in the rewritten workflow.

## Technology Decisions

### No New Dependencies
This feature adds no new runtime or dev dependencies. All tools used:
- `git` (built into CI runner)
- `npx markdownlint-cli2` (downloaded on-demand, not committed)
- `node dist/cli/program.js agent` (existing CLI)

### Markdown File Naming Convention
**Decision**: All workflow-managed documentation files use UPPERCASE names (matching existing convention):
- `docs/ARCHITECTURE.md` (renamed from `docs/architecture.md`)
- `docs/API.md`
- `docs/AUTOMATION.md`
- `docs/CONTRIBUTING.md`
- `docs/TESTING.md`
- `docs/ROUTING.md` (new)
- `docs/PROVIDERS.md` (new)
- `docs/CONFIGURATION.md` (new)
- `CHANGELOG.md` (root, already uppercase)

Unmanaged files keep their existing names (lowercase): `permission-system.md`, etc.

### Prompt Architecture
**Decision**: Two-layer prompt (simplified from current three-layer):
1. **System prompt** (`--system`): Role definition only (1-2 sentences). No detailed instructions.
2. **Message prompt** (assembled from templates): All context, requirements, scope, and instructions.

This eliminates the conflicting dual-prompt issue (Issue 6).
