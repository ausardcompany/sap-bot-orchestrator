# Contract: Template System

## Overview

The template system assembles an LLM prompt from ordered, composable markdown fragments. It supports project-specific overrides layered on top of bundled defaults.

## Resolution Priority

For each template file, the system checks:

1. **Project override**: `${inputs.templates-dir}/<filename>` — provided by the caller's repository
2. **Action default**: `${github.action_path}/templates/<filename>` — bundled with the action

If neither exists, the template is skipped (for conditional templates) or the action fails (for required templates).

## Required Templates

These must exist in either the override directory or the action defaults:

| File | Purpose |
|------|---------|
| `01-header.md` | Project context: structure, commands, tech stack. Must instruct the agent to read source code before writing. |
| `02-changed-files-header.md` | Section divider: "## Changed Files" |
| `03-commits-header.md` | Section divider: "## Recent Commits" |
| `04-diff-header.md` | Section divider: "## Code Changes" with tool-usage note |
| `05-scope-header.md` | Section divider: "## Documentation Scope" |
| `06-requirements.md` | General documentation quality requirements (accuracy, Mermaid diagrams, etc.) |
| `08-footer.md` | Output format requirements, quality checks |

## Conditional Templates

Located in `07-conditional/`. Included when matching doc files are in scope:

| Template File | Included When | Current Alexi Mapping |
|--------------|---------------|----------------------|
| `architecture-api.md` | `docs/ARCHITECTURE.md` or `docs/API.md` in scope | `cli_changed \|\| core_changed` |
| `routing.md` | `docs/ROUTING.md` in scope | `routing_changed` |
| `providers.md` | `docs/PROVIDERS.md` in scope | `providers_changed` |
| `configuration.md` | `docs/CONFIGURATION.md` in scope | `config_changed` |
| `testing.md` | `docs/TESTING.md` in scope | `tests_changed` |
| `automation.md` | `docs/AUTOMATION.md` in scope | `workflows_changed \|\| scripts_changed` |
| `changelog-contributing.md` | **Always included** | Always |

## Conditional Template Name Resolution

The action maps doc file paths to conditional template names:

```
docs/ARCHITECTURE.md → architecture  → 07-conditional/architecture.md
docs/API.md          → api          → 07-conditional/api.md
docs/ROUTING.md      → routing      → 07-conditional/routing.md
CHANGELOG.md         → changelog    → 07-conditional/changelog.md
```

Algorithm: `basename(doc_file, ".md") | lowercase`

For the Alexi project, `architecture-api.md` is a combined template. The mapping can be customized via project overrides — if a project has `07-conditional/architecture.md` in its templates-dir, it will be picked up when ARCHITECTURE.md is in scope.

## Dynamic Content Files

Generated at runtime and injected between static templates:

| File | Generated By | Content |
|------|-------------|---------|
| `analysis.md` | Step 4 (Analyze) | Changed files list with module categorization |
| `commits.md` | Step 7 (Commits) | Git log since last doc commit |
| `diff.md` | Step 8 (Diff) | Diff stats + truncated code changes per file type |
| `scope.md` | Step 6 (Scope) | List of doc files to update |

## Assembly Order

```
01-header.md
02-changed-files-header.md
  → analysis.md (dynamic)
03-commits-header.md
  → commits.md (dynamic)
04-diff-header.md
  → diff.md (dynamic)
05-scope-header.md
  → scope.md (dynamic)
06-requirements.md
07-conditional/*.md (per scope)
07-conditional/changelog-contributing.md (always)
08-footer.md
```

Output file: `prompt.md` (fed to the LLM agent via `--message-file`)

## Bundled Default Templates

The action ships with generic default templates suitable for any TypeScript/Node.js project. The Alexi project overrides these with project-specific versions in `.github/templates/`.

### Default `01-header.md` (generic)

```markdown
# Documentation Generation Task

You are a technical documentation writer. Your task is to read the source code
using the provided tools and generate/update documentation files.

## Important Instructions

1. **Always read source code** before writing documentation — use the `read` and `grep` tools
2. **Write to exact file paths** specified in the Documentation Scope section
3. **Include code examples** from the actual codebase
4. **Be accurate** — every statement must be verifiable from the source code
```

### Default `06-requirements.md` (generic)

```markdown
## Documentation Requirements

1. Each documentation file must be self-contained and understandable independently
2. Include at least one Mermaid diagram per documentation file where applicable
3. Code examples must come from the actual codebase (read files first)
4. Use consistent formatting and heading levels
5. Document public APIs, configuration options, and usage patterns
6. Include cross-references between related documentation files
```

Projects override these with domain-specific content (Alexi's `01-header.md` includes 30 `src/` directories, CLI command list, and tech stack details).
